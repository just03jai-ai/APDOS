# Runtime Orchestrator

The Runtime Orchestrator connects APDOS workflow execution to Skill Governance and Skill Runtime.

It allows a workflow stage to determine:

- which agent owns the stage
- which governed skills should run
- which artifacts are generated by skill execution
- which workflow stage should run next

## Runtime Flow

```text
Workflow Engine
  -> Runtime Stage Resolution
  -> Runtime Health Validation
  -> Skill Governance
  -> Skill Runtime
  -> Artifact Generation
  -> Workflow Stage Completion
```

## Services

### RuntimeStageResolver

Supports:

- `resolveWorkflowStage()`
- `resolveNextStage()`

It reads workflow instances from the Workflow Engine and selects the running stage first, or the next pending stage.

### RuntimeAgentResolver

Supports:

- `resolveAgentForStage()`

It uses Skill Governance stage mappings to select the owning agent for a workflow stage.

### RuntimeSkillResolver

Supports:

- `resolveSkillsForStage()`

It uses Skill Governance metadata to select all enabled skills mapped to a workflow stage.

### RuntimeSkillExecutor

Supports:

- `executeSkills()`

It executes resolved skills through the Skill Runtime using deterministic execution.

Within a stage, artifacts are propagated between skills in execution order:

```text
currentArtifacts = stageInputArtifacts

for skill in governedSkills:
  result = executeSkill(skill, currentArtifacts)
  currentArtifacts += result.artifacts
```

This lets later skills consume artifacts generated earlier in the same stage.

### RuntimeArtifactGenerator

Supports:

- `createArtifactsFromExecution()`

It collects artifacts returned by Skill Runtime execution and can register them in the Artifact Registry.

### RuntimeOrchestratorService

The facade service coordinates all runtime steps and returns:

- `selectedAgent`
- `executedSkills`
- `generatedArtifacts`
- `nextStage`

If skill execution or artifact creation fails after a stage has started, the orchestrator marks the workflow stage as failed so the workflow does not remain running.

### RuntimeHealthService

Supports:

- `validateRuntime()`
- `validate()`

It checks:

- governed skills are present in the Skill Runtime registry
- the runtime registry has no extra ungoverned skill names
- Skill Governance dependencies are valid
- governed skills can be loaded by the Skill Runtime

## Example

```ts
const result = await runtimeOrchestrator.executeSkills({
  workflowId: "workflow:1",
  inputArtifacts: [discoveryReport]
});

console.log(result.selectedAgent);
console.log(result.executedSkills.map((execution) => execution.skill.skillId));
console.log(result.generatedArtifacts);
console.log(result.nextStage);
```

## Current Limits

The orchestrator uses deterministic mock Skill Runtime execution. It does not execute real prompts or external LLM calls.
